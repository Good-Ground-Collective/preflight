# preflight — ESLint charter-enforcement plugin

## Problem Statement

OUr code review sessions are being clogged up by nit-pick comments that should ideally be resolved before a pr is opened, allowing us to focus on larger more structural issues in the code review session itself

## Solution

Build `preflight`, an ESLint plugin for TypeScript that mechanically enforces our coding charter so nit-level violations are caught before a PR is opened. It ships:

- 8 custom rules (`no-loose-functions`, `no-planning-identifiers`, `no-paragraph-comments`, `no-throw-helpers`, `no-switch-with-nested-if`, `error-class-sets-name`, `service-shape`, `constructor-single-props`)
- config-only wiring of stock `@typescript-eslint` / `unicorn` rules (member-ordering, naming-convention, filename-case, and recommended-only niceties)
- two rulesets: `go-no-go` (deterministic blocking gate, all `error`) and `recommended` (⊇ go-no-go, adds the fuzzier rules + niceties)

## Background

The charter already exists but nothing enforces it. It's largely based off my personal preferences, but it;s hard to begin enforcement if the enforcement is opionated. We need something determinisitc that engineers can run

## Technical Notes

- Flat-config-first ESLint plugin (ESLint 9+), exporting `rules`, `configs['go-no-go']`, `configs.recommended`.
- Custom rules are AST-based and avoid type-info where possible (fast, works without `parserServices`). `service-shape`, `constructor-single-props`, and `error-class-sets-name` may need light type info — flagged per-rule.
- Rule options are user-configurable where noted (e.g. `no-loose-functions` suffix allowlist, `no-planning-identifiers` token list).

## Known Gaps and Edge Cases

- **Autofix scope for v1:** `fix` for `error-class-sets-name`, `suggest` for `no-planning-identifiers`, everything else report-only. Autofix on the ambiguous rules (loose-functions, paragraph-comments, throw-helpers, switch-nested-if, service-shape, constructor-single-props) is explicitly out of scope — no deterministic transform exists. Config-only stock rules inherit whatever autofix upstream provides.
- `service-shape` / `constructor-single-props` are false-positive-prone — hence recommended-only, never in the blocking gate.
- Config-only stock rules assume `@typescript-eslint` and `unicorn` are peer deps in the consumer's project.

## Ticket Sketch

- **Scaffold the plugin** — package setup, flat-config exports (`rules`, `configs`), `RuleTester` harness, CI. *(first — everything depends on it)*
- **`no-loose-functions`** — module-level function ban with configurable suffix allowlist
- **`no-planning-identifiers`** — configurable token/pattern ban in comments + strings, with `suggest` to remove
- **`no-paragraph-comments`** — file-top / non-declaration paragraph-comment ban
- **`no-throw-helpers`** — flag single-`throw` functions
- **`no-switch-with-nested-if`** — flag `if` inside `switch` cases
- **`error-class-sets-name`** — require `name` on Error subclasses, with autofix
- **`service-shape`** — flag interface+default / factory-closure deviations (recommended-only)
- **`constructor-single-props`** — flag multi-param constructors (recommended-only)
- **Assemble rulesets** — wire the 8 custom rules + stock `@typescript-eslint`/`unicorn` config into `go-no-go` and `recommended`; verify `recommended ⊇ go-no-go`. *(last — depends on all rules existing)*

Ordering only matters at the ends (scaffold first, rulesets last); the rule tickets in between are parallelizable.
